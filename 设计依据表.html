一、概念结构设计依据

设计要点设计依据
统一身份 + 子类(ISA)AppUser 统一登录鉴权，教师/学生专属属性下沉到 Teacher / Student，避免一张大表堆空列；用 1:1（UserID 唯一）保证一个账号恰好一条角色信息。
选课用 m:n（CourseStudent）一个学生选多门课、一门课含多名学生，拆为关联表。
报告用关联实体 Report报告自带文件、成绩、评语等属性；UQ(LabID, SNO) 实现“每生每实验项目一份”。
评阅单独建联系（GradedBy）提交人（学生）≠ 批改人（教师），分开建模以记录批改责任。
审计弱关联AuditLog.UserID 可空且不设外键，用户被删后日志仍保留，便于追溯。

二、逻辑结构设计依据（关系模式 / 3NF）

设计要点设计依据
E-R 转换规则1 端主键放到 n 端做外键；m:n 单独建关联表（复合主键）。
满足 3NF非主属性只依赖主键、无传递依赖——课程不存教师院系、报告不存学生姓名/实验名（靠外键 JOIN 取），消除冗余与更新异常。
代理键优先多数表用 IDENTITY 整型主键（稳定、连接快）；学号 SNO 是稳定自然键，直接做主键。
约束落到数据层CHECK（角色枚举、状态枚举、成绩 0~100）+ UNIQUE（用户名、(LabID,SNO)、(CourseID,LabName)）防止业务规则被绕过。
级联防“多路径级联”仅 Course→LabProject/CourseStudent、LabProject→Report 级联删除；Student/Teacher 不级联到 Report（SQL Server 禁止多路径级联 + 保护历史成绩），删除前由应用层校验。

三、物理结构设计依据（索引）

设计要点设计依据
主键聚簇索引IDENTITY 单调递增，点查最快、插入无热点页分裂。
唯一索引一举两得Username、(LabID,SNO) 等既保证唯一，又加速等值查找。
外键必须手建索引SQL Server 不自动为外键建索引，故对高频连接/过滤的 Course.TeacherID、Report.SNO、CourseStudent.SNO 显式建索引，避免全表扫描。
组合索引列序遵循“等值列在前、排序/范围列在后”：IX_LabProject_CourseStatus(CourseID, Status)、IX_Report_LabSubmitted(LabID, SubmittedAt DESC)。
降序索引对齐 ORDER BY DESC报告列表、审计日志都“最新在前”，降序索引免去排序算子。
写放大权衡只为高频读查询建索引；低选择性列（Role/Status 单列）与低频统计不单独建索引。
